## 1. 아키텍처 평가

### 1-1. 컴포넌트 구조 -- 전반적으로 건전하나 핵심 결함 존재

| 발견 | 심각도 | 상세 | 수정 방안 |
|------|--------|------|-----------|
| **L1 만세력 자체 구현이 여전히 TECH_STACK에 남아있음** | CRITICAL | DESIGN.md Decision Audit Trail에서 "만세력 외부 소스 전환 수용"이 승인되었으나, 기술스택_TECH_STACK.md의 Section 2 "사주 연산 엔진"에는 여전히 "양력→음력 변환 (자체 변환 테이블 구현)", "24절기 절입시각 (한국천문연구원 데이터)" 등 **자체 구현 전제**로 기술되어 있다. 문서 간 불일치가 CRITICAL하다. | 기술스택 문서를 외부 소스 전환으로 즉시 동기화. L1의 역할을 "외부 만세력 소스 호출 + 결과 정규화"로 재정의. |
| **웹 랜딩페이지와 모바일 앱의 백엔드 공유 전략 부재** | HIGH | CEO 리뷰에서 "웹 랜딩페이지 Phase 1 추가"가 승인되었으나, 이 웹 랜딩이 기존 Rust API를 공유하는지, 별도 서버인지, 인증은 어떻게 되는지 아무런 설계가 없다. | 아래 섹션 7에서 상세 제안. |

### 1-2. 커플링 분석

| 커플링 | 수준 | 평가 |
|--------|------|------|
| App ↔ Rust API | REST + SSE, 느슨함 | 양호. API 버전닝 전략 필요 (v1 prefix). |
| Rust API ↔ Claude | reqwest HTTP, trait 추상화 | 양호. LLMProvider trait으로 폴백 가능. |
| Rust API ↔ NanoBanana | ImageGenerator trait 추상화 | 양호. 폴백(텍스트 카드) 설계됨. |
| Rust API ↔ RevenueCat | 직접 HTTP + webhook | **주의**. RevenueCat 다운 시 내부 orders 테이블로 폴백은 좋으나, webhook 수신 실패 시 재시도 로직 필요. |
| Rust API ↔ 만세력 외부 소스 | **미정의** | **CRITICAL**. 외부 소스 결정 전까지 인터페이스 정의 불가. ManseryeokProvider trait을 먼저 정의하고, mock 구현으로 개발 시작해야 함. |
| 웹 랜딩 ↔ Rust API | **미정의** | 웹이 같은 API를 직접 호출할 경우 CORS + 인증 전략 필요. |

### 1-3. 웹 랜딩페이지 + 모바일 앱 공존 전략

현재 설계에서 누락된 핵심 질문들:

1. **인증 분기**: 웹 방문자는 로그인 없이 무료 카드 생성 가능해야 하는가? (앱에서는 기기당 rate limit)
2. **API 공유**: 웹 → Rust API 직접 호출 시 CORS 허용 도메인 관리 필요
3. **SEO와 동적 콘텐츠**: 공유된 카드 URL에 OG image를 동적으로 생성해야 함 (Next.js ISR 또는 Edge Function)
4. **결제 플로우**: 웹에서는 IAP 불가. 웹은 순수 유입용(무료 카드 + 앱 설치 유도)인지, 웹 결제도 지원할 것인지?

## 2. 에지케이스 -- 10x 부하 시 무엇이 깨지는가

### 2-1. 부하 관련 에지케이스

| 시나리오 | 깨지는 지점 | 심각도 | 수정 방안 |
|----------|-------------|--------|-----------|
| **바이럴 폭발 -- 무료 카드 10x (1일 10만 건)** | NanoBanana API rate limit/timeout. 동일 생년월일 캐싱이 있으나, 생년월일 조합이 ~36만개(100년 x 365일 x 1성별, 시진 제외)이므로 캐시 히트율이 낮다. 시진 포함 시 ~432만 조합. | HIGH | NanoBanana에 rate limit 확인 필수. 큐 기반 비동기 생성 + 폴백 텍스트 카드. 캐시 키에 시진을 포함하되, "오늘의 키워드"가 날짜 의존이면 캐시 무효화 주기 설계 필요. |
| **Claude API 동시 요청 100건 이상** | Anthropic API의 기본 rate limit (RPM, TPM). 유료 상담 + 오늘의 운세 + 채팅이 모두 Claude를 공유. | HIGH | Claude API tier별 rate limit 사전 확인. 오늘의 운세는 **배치 사전 생성** (일주별 60종 x 일간 조합, 새벽 크론잡). 실시간 Claude 호출은 유료 해석 + 채팅만으로 한정. |
| **Fly.io 2인스턴스 한계** | SSE 커넥션이 인스턴스에 고정(sticky). 로드밸런서가 SSE 중 인스턴스 재배포하면 스트림 끊김. | MEDIUM | SSE 재연결 로직 (Last-Event-ID 기반). Flutter 클라이언트에서 자동 재연결 + 서버에서 이벤트 ID로 이어서 전송. |
| **Supabase 무료 티어 DB 커넥션 한계** | Supabase 무료는 동시 커넥션 ~60개. Rust sqlx 커넥션 풀 + Worker 합산 시 초과 가능. | MEDIUM | 초기부터 Supabase Pro ($25/월) 사용 또는 PgBouncer(Supabase 내장) 활성화. sqlx 풀 max_connections를 명시적으로 설정 (기본값이 아닌 10-20으로). |
| **Redis 메모리 초과** | HMAC 캐시 TTL 30일 + 작업큐 + 세션 mutex. 사용자 증가 시 메모리 부족. | LOW | Upstash Redis 사용 시 자동 스케일. maxmemory-policy: allkeys-lru 설정. 캐시와 큐를 별도 Redis 인스턴스로 분리 (Phase 2). |

### 2-2. Nil/Empty/Error 경로

| 경로 | 현재 처리 | 누락된 케이스 | 심각도 |
|------|-----------|---------------|--------|
| **출생시간 "모름"** | 시주 제외 계산, 배너 표시 | 시주 없는 사주의 L2 검증 로직이 정의되지 않음. 오행 배분에서 시주 2글자(천간+지지) 빠지면 점수 체계 자체가 달라짐. "모름" 케이스의 golden test가 별도로 필요. | HIGH |
| **음력 날짜 유효성** | 인라인 에러 표시 | 외부 만세력 소스가 반환하는 에러와 앱 에러 메시지 매핑이 미정의. 외부 API timeout vs invalid date 구분 필요. | MEDIUM |
| **채팅 50턴 정확히 도달** | 만료 처리 | 50번째 턴에서 Claude 응답이 길 경우 (토큰캡 도달 전 SSE 진행 중) 중간 절단 UX 미정의. 마지막 턴은 "이것이 마지막 답변입니다" 안내 후 전송해야 함. | MEDIUM |
| **72시간 세션 중 서버 재배포** | 미정의 | Fly.io 배포 시 기존 SSE 연결 끊김. 진행 중인 채팅 세션의 컨텍스트(최근 20턴)가 메모리에만 있으면 유실. | HIGH |
| **결제 성공 후 앱 강제 종료** | 푸시 알림으로 복귀 유도 | Worker가 이미지 생성 중 크래시하면? 아티팩트별 체크포인트가 TODOS.md [P2]에 있으나, Phase 1에서 미구현 시 **결제했는데 결과 못 받는** 최악 시나리오. | CRITICAL |
| **동일 사용자 다중 기기 로그인** | 미정의 | A 기기에서 결제 후 B 기기에서 결과 조회 시 세션 처리. JWT는 기기별 발급되나 세션은 사용자별. | MEDIUM |

## 5. 숨겨진 복잡성 -- "단순해 보이지만 아닌 것"

### 5-1. [CRITICAL] 오늘의 운세 -- 비용 폭탄 가능성

**설계**: "생년월일 기반 일간 운세 텍스트 (Claude API, 이미지 생성 없음 -- 비용 최소화)"

**숨겨진 복잡성**:
- 사용자별 개인화된 운세 = 사용자 수 x Claude API 호출. MAU 1만명이 매일 운세를 보면 월 30만 건의 Claude 호출.
- 비용 추정: 입력 ~500 tokens (사주 데이터 + 날짜) + 출력 ~300 tokens = 약 500 x $15/M + 300 x $75/M = $0.03/건. 30만건 x $0.03 = **$9,000/월 (약 1,200만원)**.
- DESIGN.md의 추정 "1만 x ~50원 = ~50만원/월"과 **24배 차이**. 50원 추정은 MAU가 아닌 DAU 기준이거나, 토큰 계산이 과소.

**수정 방안**:
1. 운세를 **일주(日柱) 기반으로 그룹화** -- 일주는 60종(육십갑자). 매일 60종의 운세만 사전 생성하면 모든 사용자 커버.
2. 사전 생성 비용: 60건 x 30일 x $0.03 = **$54/월** (7만원). 현재 추정 대비 1/170.
3. 추가 개인화는 클라이언트에서 사주 프로필 데이터와 조합 (템플릿 기반).

### 5-2. [HIGH] 채팅 히스토리 압축 -- LLM 호출 비용

**설계**: "오래된 턴은 요약하여 압축 (요약 로직은 Claude API로 처리)"

**숨겨진 복잡성**:
- 20턴 초과 시마다 요약 생성 = 추가 Claude 호출. 50턴 세션에서 요약이 3회 발생 가능.
- 요약 자체가 비용이고, 요약 품질이 나쁘면 후속 답변 품질 저하.
- 요약 실패 시 채팅이 멈추는 SPOF.

**수정 방안**:
1. 요약에 Claude Haiku 사용 (비용 1/30). 요약은 정확성보다 핵심 추출이 목적이므로 충분.
2. 요약 실패 시 최근 20턴만 유지 (요약 없이). 품질은 약간 떨어지나 서비스 중단 방지.
3. 요약 결과를 DB에 저장하여 같은 요약을 재계산하지 않음.

### 5-3. [HIGH] NanoBanana 이미지 일관성

**설계**: "스타일 시드 고정값 결정"으로 일관성 확보

**숨겨진 복잡성**:
- AI 이미지 생성은 시드가 같아도 프롬프트 미세 변경에 스타일이 크게 달라질 수 있음.
- 같은 사용자의 5-8장 결과 이미지가 서로 다른 스타일로 나오면 UX 품질 저하.
- 캐릭터(월하선생) 일관성 유지가 텍스트 프롬프트만으로는 어려움.

**수정 방안**:
1. NanoBanana의 reference image 기능 활용 (있다면). 기준 이미지를 고정하고 변형 생성.
2. 이미지 프롬프트를 극도로 구조화 -- 변하는 부분(오행 색상, 주제)과 고정 부분(캐릭터, 배경 스타일)을 명확히 분리.
3. 생성된 이미지의 품질 검증 자동화 (이미지 유사도 score, 임계값 미달 시 재생성).

### 5-4. [HIGH] SSE 스트리밍 -- 모바일 환경 불안정성

**설계**: L3 해석을 SSE로 실시간 스트리밍

**숨겨진 복잡성**:
- 모바일 환경에서 SSE는 Wi-Fi↔셀룰러 전환, 터널/지하철, 화면 잠금 등에서 빈번하게 끊김.
- Flutter의 SSE 라이브러리(`eventsource` 또는 `dio` 기반)가 자동 재연결을 안정적으로 처리하는지 검증 필요.
- 재연결 시 "이미 수신한 부분부터 이어서" 보내려면 서버가 이벤트 ID 기반 재전송을 지원해야 함.

**수정 방안**:
1. 서버: 각 SSE 이벤트에 순차 ID 부여. 클라이언트의 `Last-Event-ID` 헤더 지원.
2. 클라이언트: 수신된 텍스트를 로컬 저장. 재연결 시 서버에 마지막 ID 전송.
3. 최악의 경우(재연결 실패): 완성된 결과를 REST API로 폴링 (SSE는 UX 향상용, REST가 신뢰성 보장용).
4. 이미지 생성 완료 알림은 SSE보다 FCM 푸시가 더 신뢰성 높음 (앱 백그라운드 대응).

### 5-5. [MEDIUM] Firebase Dynamic Links 서비스 종료

**설계**: "SDK: Firebase Dynamic Links 또는 Kakao Link SDK"

**숨겨진 복잡성**:
- Firebase Dynamic Links는 **2025년 8월 25일에 서비스 종료**되었다. 현재(2026-03-24) 이미 종료된 서비스를 사용할 수 없다.

**수정 방안**:
1. 대안: Adjust, Branch.io, 또는 자체 Universal Links/App Links 구현.
2. 카카오 Link SDK는 카카오톡 공유에는 적합하나, 범용 딥링크 솔루션이 아님.
3. 자체 구현 권장: Apple Universal Links + Android App Links를 직접 설정. 웹 랜딩페이지가 있으면 폴백 라우팅도 자연스러움.

### 5-6. [MEDIUM] Supabase → AWS 마이그레이션 경로

**설계**: "Supabase 무료 → Supabase Pro → AWS RDS"

**숨겨진 복잡성**:
- Supabase의 Row Level Security(RLS), Realtime, Storage 등을 사용하면 AWS 전환 시 모두 재구현 필요.
- Supabase를 "단순 PostgreSQL 호스팅"으로만 쓰면 전환 쉬우나, 그러면 Supabase 쓰는 이유가 없음.

**수정 방안**:
- Supabase는 PostgreSQL 호스팅 + pgBouncer로만 사용. RLS, Realtime, Auth 등 Supabase 전용 기능 사용 금지.
- 인증은 Rust 서버 자체 JWT. Storage는 S3 호환 스토리지.
- 이렇게 하면 향후 AWS RDS로 pg_dump/pg_restore만으로 전환 가능.

### 6-3. Rust 통합 방법

**방법 A: 한국천문연구원 API 직접 호출 (권장)**
```

    fn get_jeolgi_time(year: u16, jeolgi: Jeolgi) -> Result<DateTime>;

// 구현 3: CachedProvider (Redis/DB 캐시 래퍼)
```

**방법 B: Python 마이크로서비스 (대안)**
- Python `korean-lunar-calendar` + 사주 계산 로직을 FastAPI 서비스로 분리.
- Rust에서 HTTP 호출. 지연시간 +5-10ms (로컬 통신).
- 장점: Python 생태계 활용, 빠른 구현. 단점: 서비스 하나 더 운영.

**권장: 방법 A** -- Rust 서버 내에서 외부 API 호출 + 결과 캐싱. Python 서비스 추가 운영 부담 회피. 캐시 히트율이 높아지면 외부 API 의존도 자연 감소.

## 9. 결론 및 최우선 조치 항목

이 설계는 **전반적으로 사려 깊고 에지케이스에 대한 인식이 높다**. 특히 3-layer 사주 엔진 분리, 인터랙션 상태 테이블, progressive reveal UX, 오프라인 처리 등은 잘 설계되어 있다. 그러나 아래 5가지가 구현 전 반드시 해결되어야 한다:

1. **만세력 외부화 구체 확정 + 문서 동기화** -- ManseryeokProvider trait 정의, 한국천문연구원 API 테스트, golden test 100건 확보.
2. **오늘의 운세 비용 모델 수정** -- 사용자별 실시간 생성이 아닌 일주 60종 배치 생성 방식으로 전환.
3. **Claude API 비용 시뮬레이션** -- 유료 세션당 총 토큰 시뮬레이션 (해석 + 채팅 50턴 + 요약 3회). per-session 비용캡 구체 수치 확정.
4. **결제 후 장애 복구 최소 보장** -- Worker 크래시 시 주문 상태 기반 자동 재시도. Phase 1에서 아티팩트 체크포인트의 최소 버전 구현.
5. **Firebase Dynamic Links 대체** -- 이미 종료된 서비스. Universal Links/App Links 자체 구현으로 전환.

이 5가지 해결 후 구현 착수를 권장한다.
