## 1. 올바른 문제를 풀고 있는가? 10배 임팩트를 위한 리프레이밍

### 현재 프레이밍
"청월당보다 싸고, 대화가 되는 AI 사주 상담 앱"

### 문제점 [심각도: HIGH]
이 프레이밍은 **청월당의 열화판**을 만들겠다는 선언이다. "더 싸고 대화 가능"은 기능 차별화이지 카테고리 차별화가 아니다. 청월당은 이미 월간 100만 사용자, CES 2025 참가, 스탠퍼드 만족도 9.67/10이라는 브랜드 해자를 갖고 있다. 가격을 절반으로 내리는 것은 "프리미엄이 아니다"라는 시그널을 보내는 것이다.

### 리프레이밍 제안
**"사주를 매일 쓰는 생활 도구로 만들기"**

현재 사주 시장의 근본 문제는 1회성이라는 것이다. 태어난 날이 바뀌지 않으므로 한 번 보면 끝이다. 이것은 DESIGN.md에서도 "사주는 1회성 상품(생년월일 불변). 반복 수익 없으면 LTV 낮음"이라고 정확히 인식하고 있다. 그런데도 v1은 1회성 결제 모델로 출발한다.

리프레이밍하면:
- **1회성 사주 상담이 아닌, "매일 대운/세운/월운이 바뀌는 나의 사주 대시보드"**
- 오늘의 운세는 리텐션 훅이 아니라 **핵심 상품 그 자체**
- 구독 모델이 v2가 아니라 v1의 핵심
- 월 4,900원 구독 1만명 = 월 4,900만원 (IAP 수령 3,430만원) vs 1회 15,000원 x 1,000건 = 월 1,050만원

**이 리프레이밍만으로 LTV가 3-5배 늘어난다.**

### 조치
구독 모델을 v2에서 **v1 핵심**으로 승격. 1회성 상담은 구독 전환 리드마그넷으로 재정의.

## 2. 전제(Premises) 검증 — 명시된 것과 숨겨진 것

### 명시된 전제 5개 중 위험한 것

| # | 전제 | 상태 | 위험도 |
|---|------|------|--------|
| P1 | 가격 1-1.5만원이 진입장벽을 낮춤 | **틀릴 수 있음** | HIGH |
| P2 | NanoBanana Lottie풍 이미지가 차별화 | **틀릴 수 있음** | CRITICAL |
| P3 | 콘텐츠 품질이 구매 결정의 핵심 | 맞음 | — |
| P4 | 대화형 AI가 일방향보다 높은 만족도 | **미검증** | HIGH |
| P5 | 이미지 공유가 유입에 기여 | **미검증** | MEDIUM |

**P1이 틀릴 수 있는 이유** [HIGH]:
청월당 3.7만원은 "프리미엄이라 비싸다"가 아니다. 사주를 보겠다고 결심한 사람에게 3.7만원은 이미 수용 가능한 가격이다 (오프라인 사주카페 3-10만원). 15,000원으로 내리면 "싸니까 별로 아닐까?"라는 역효과 가능. 가격 민감 사용자는 무료 앱(점신, 포스텔러)으로 가지, 15,000원을 내지 않는다. **이 가격대는 아무도 없는 No Man's Land일 수 있다.**

**P2가 틀릴 수 있는 이유** [CRITICAL]:
NanoBanana로 생성한 "Lottie풍 파스텔 벡터 일러스트"가 청월당의 웹툰 캐릭터 기반 스토리텔링을 이길 수 있는가? 청월당은 '청월아씨', '홍연아씨', '월하소녀' 등 고유 캐릭터로 감정적 유대를 만든다. AI 생성 이미지는 매번 미묘하게 다르고, 캐릭터 일관성이 없다. **이미지가 차별화가 아니라 약점이 될 수 있다.**

### 숨겨진(암묵적) 전제 — 가장 위험한 것들

| # | 숨겨진 전제 | 위험도 |
|---|-----------|--------|
| H1 | "1인 개발자가 Rust 만세력 엔진을 정확하게 구현할 수 있다" | **CRITICAL** |
| H2 | "사용자가 AI 사주를 '진짜'로 느낄 것이다" | **HIGH** |
| H3 | "Flutter + Rust 조합으로 앱스토어 심사에 문제없다" | MEDIUM |
| H4 | "Claude API 가격이 현재 수준을 유지할 것이다" | MEDIUM |
| H5 | "앱 전용(웹 없음)으로도 충분한 초기 유입이 가능하다" | **HIGH** |

**H1 상세** [CRITICAL]:
만세력은 단순한 달력이 아니다. 양력-음력 변환, 24절기 절입시각(분 단위), 야자시/조자시, 서머타임(1948-1988), 윤달 처리, 대운 순행/역행 등 에지케이스의 바다이다. TODOS.md에서도 "만세력 달력 엣지케이스 전체 정의서 작성"이 P1으로 올라와 있고, "Codex 2회 연속 지적"이라고 적혀 있다. 명리학 전문가 없이 1인 개발자가 이것을 정확하게 구현하는 것은 **사실상 불가능**하다. 하나라도 틀리면 유료 고객의 신뢰가 즉시 붕괴한다.

**조치**: 만세력 엔진을 자체 구현하지 말고, 검증된 만세력 라이브러리(Python lunar_calendar, 한국천문연구원 API)를 Rust FFI로 래핑하거나, L1을 Python 마이크로서비스로 분리하라. "Rust 네이티브 3-layer"라는 아키텍처적 미학보다 **정확성**이 10배 중요하다.

**H5 상세** [HIGH]:
"Approach C: 하이브리드"를 기각한 사유가 "사용자가 모바일 퍼스트를 선택"이라고만 되어 있다. 그런데 청월당은 웹(Next.js SSG)이고 성공했다. 앱 전용은 초기 유입의 병목이다. 앱스토어에서 "사주" 검색 시 점신, 포스텔러가 이미 상위 장악. 신규 앱이 ASO만으로 올라가는 것은 비현실적이다. **웹 랜딩페이지 없이 앱 전용으로 시작하면 초기 유입 확보가 극도로 어렵다.**

## 4. 충분한 분석 없이 기각된 대안들

### 4-1. 웹 퍼스트 (Approach A) 기각 재검토 [HIGH]
기각 사유: "GIF 파이프라인 복잡성, AI 이미지 비용(200원/장), Vercel 타임아웃"

이 기각 사유는 모두 **NanoBanana 채택 전의 분석**이다. NanoBanana가 27원/장이고 GIF를 네이티브 지원한다면, Approach A의 기각 사유 3개 중 2개가 해소된다. 웹 퍼스트로 빠르게 수요 검증 → 모바일 전환이 **훨씬 낮은 리스크** 경로였다. 적어도 웹 랜딩페이지 + 무료 카드 생성은 Phase 1에 포함해야 한다.

### 4-2. 만세력 외부 서비스 활용 [CRITICAL]
만세력 API나 검증된 라이브러리를 쓰는 대안이 아예 논의되지 않았다. "Rust 네이티브 3-layer 엔진"이 기술적 자존심에서 나온 결정이 아닌지 검증 필요. 한국에는 여러 만세력 API가 존재한다 (만세력닷컴, 한국천문연구원 공공데이터).

### 4-3. 기존 플랫폼 입점 [MEDIUM]
카카오톡 미니앱, 카카오 채널, 네이버 스마트스토어 등 기존 플랫폼에 입점하여 초기 유입을 확보하는 대안이 전혀 논의되지 않았다. 자체 앱 개발 전에 카카오 채널에서 무료 미니 사주를 제공하여 수요를 검증하는 것이 훨씬 빠르다.

### 4-4. Claude API 대신 더 저렴한 모델 [MEDIUM]
Opus 4.6은 가장 비싼 Claude 모델이다 (입력 $15/MTok, 출력 $75/MTok). Sonnet 4나 Haiku 4가 사주 해석에 충분한지 테스트한 흔적이 없다. L3의 역할이 "수치를 자연어로 번역"이라면 Sonnet으로 충분할 가능성이 높다. **비용이 5-10배 절감**될 수 있다.

## 5. 경쟁 리스크 — 누가 먼저/더 잘 풀 수 있는가

### 직접 위협 [CRITICAL]

**청월당(로켓AI)**이 가장 큰 위협이다.
- 이미 월간 100만 사용자, 강력한 브랜드
- 기술력 보유 (Next.js SSG, AI 이미지 생성 파이프라인)
- AI 채팅 기능 추가는 **몇 주면 가능** — 청월당 랜딩페이지에 이미 "청월아씨에게 궁금한 점을 물어보세요" 자유 질문 기능이 존재
- 당신이 6개월간 Rust 엔진을 만드는 동안, 청월당은 기존 인프라에 채팅을 붙이면 끝

**점신(테크랩스)**도 위협이다.
- 매출 830억, 1,500만 다운로드의 규모
- AI 채팅 기능 추가에 필요한 자원이 충분
- 기존 사용자 기반에 업셀이 가능

### 방어선 분석
문서에서 주장하는 차별화:
1. "3-layer 사주 엔진 정확성" — 구현에 수개월, 검증에 전문가 필요. 경쟁사는 이미 검증된 엔진 보유
2. "대화형 AI 채팅" — 기술적 해자 없음. ChatGPT wrapper로 며칠이면 구현 가능
3. "NanoBanana 이미지" — 청월당의 커스텀 웹툰 캐릭터 대비 약점

**모트(해자)가 사실상 없다.** 핵심 차별화로 주장하는 3가지가 모두 경쟁사가 빠르게 복제하거나 이미 더 잘하고 있다.

### 조치
해자를 만들 수 있는 유일한 경로: **데이터와 커뮤니티**. 사용자의 사주 상담 이력, 질문 패턴, 만족도 데이터가 쌓이면 프롬프트와 해석 품질이 올라간다. 하지만 이것은 규모가 있어야 작동하므로, **빠른 출시**가 최우선이다.

## 6. 수익 모델 — IAP 15,000원 가격 검증

### 가격 검증 상태 [CRITICAL]
**검증되지 않았다.** 문서 어디에도 가격 테스트, 사용자 인터뷰, WTP(Willingness-to-Pay) 조사 결과가 없다.

### 가격대 분석

| 가격대 | 경쟁자 | 사용자 심리 |
|--------|--------|------------|
| 무료~1,000원 | 점신, 포스텔러 | "가볍게 재미로" |
| 3,000~5,000원 | — (빈 구간) | "커피 한 잔 가격" |
| 15,000원 | **이 서비스** | "진지하게 사주를 보겠다" |
| 37,000원 | 청월당 | "프리미엄 경험" |

15,000원은 "재미로 해보기엔 비싸고, 진지하게 보기엔 청월당이 더 나은" 어중간한 위치이다.

### 핵심 문제
- **IAP 30% 수수료 후 수령액 10,500원** — 여기서 Claude API 비용(~500원), NanoBanana 비용(~200원) 차감하면 실질 마진 ~9,800원. 마진 자체는 나쁘지 않다
- 그러나 **문제는 볼륨**이다. 월 1,000건 유료 전환이 성공 기준인데, 이를 위해 MAU 1만명이 필요하고 (전환율 10% 가정), 앱 전용에서 MAU 1만명을 달성하기가 매우 어렵다
- **1회성 결제의 LTV 한계**: 15,000원 x 평균 1.2회 = LTV 18,000원. CAC가 5,000원만 넘어도 마케팅 ROI가 빠듯해진다

### Claude API 비용 미확정 [CRITICAL]
TODOS.md에 "[P1] Claude API 기반 비용 모델 재산정"이 있고, "기존 마진 97% 계산 전부 무효"라고 적혀 있다. **수익 모델의 근간이 아직 확정되지 않은 상태에서 가격을 확정한 것은 순서가 틀렸다.**

특히 오늘의 운세(무료)의 Claude API 비용: 1만 DAU x ~50원 = 월 50만원. 5만 DAU면 월 250만원. 무료 기능이 비용의 30%를 차지한다. Opus 4.6 대신 Haiku로 오늘의 운세를 처리하면 비용이 1/20로 줄어든다.

### 조치
1. Claude API 비용 모델을 **즉시** 확정 (이것 없이 다른 의사결정은 무의미)
2. 가격 A/B 테스트를 Phase 1에 포함 (9,900원 vs 15,000원 vs 19,900원)
3. 구독 모델(월 4,900원)을 v1에서 제공하여 LTV 극대화
4. 오늘의 운세는 Haiku/Sonnet으로 처리 (Opus는 유료 해석에만)

| ASO 경쟁 열위 | CRITICAL | "사주" 키워드는 점신/포스텔러가 장악. 신규 앱이 상위 노출되려면 6개월+ 소요 |
| 웹 유입 경로 없음 | HIGH | 앱 전용이므로 SEO 유입 불가. 청월당은 꿈해몽, 유명인 사주 등 SEO 콘텐츠로 유입 확보 |
| 광고 예산 미정 | HIGH | CAC 추정이 없다. 사주 앱의 CPI(Cost Per Install)는 보통 2,000-5,000원 |
| 바이럴 루프 미검증 | MEDIUM | "카드 공유 → 설치" 루프의 전환율 추정이 없다. 앱 설치 허들로 전환율 극도로 낮을 가능성 |
| 인플루언서/커뮤니티 전략 부재 | MEDIUM | 타로/사주 인플루언서 협업, 더쿠/에브리타임 바이럴 등 구체적 계획 없음 |

### 조치
1. **웹 랜딩페이지(무료 카드 생성)를 Phase 1에 포함** — URL 공유 가능, SEO 유입 가능
2. 초기 3개월 마케팅 예산 확정 (최소 월 200만원, CPI 3,000원 가정 시 월 667건 설치)
3. 런칭 전 타로/사주 인플루언서 3-5명과 협업 계획 수립
4. 카카오 채널에서 무료 미니 사주 제공 → 앱 설치 유도 채널 확보

## 8. 기술 리스크 평가

### 8-1. Rust 사주 엔진 [CRITICAL]

| 리스크 | 심각도 | 상세 |
|--------|--------|------|
| 만세력 정확도 | CRITICAL | 양력-음력 변환, 절입시각, 서머타임, 윤달 등 에지케이스가 수백 개. 1인 개발자가 정확하게 구현하기 극도로 어렵다 |
| 개발 기간 | HIGH | Layer 1만 2-3개월 소요 예상. Golden test 50건으로는 부족 (최소 500건 필요) |
| 전문가 부재 | HIGH | "명리학 전문가 1회 검증"만으로는 부족. 지속적 자문 필요 |
| Rust 생산성 | MEDIUM | Rust는 정확하지만 느리다 (개발 속도 기준). 1인 팀에서 Rust + Flutter 동시 개발은 부담이 크다 |

**조치**:
- 만세력 계산은 검증된 외부 소스 활용 (한국천문연구원 API 또는 검증된 라이브러리)
- L1을 별도 마이크로서비스로 분리하여 언어에 구애받지 않게 설계
- Golden test를 50건이 아닌 500건 이상으로 확대
- 명리학 전문가를 자문위원으로 확보 (1회가 아닌 지속적)

### 8-2. NanoBanana 의존성 [HIGH]

| 리스크 | 심각도 | 상세 |
|--------|--------|------|
| 서비스 안정성 | HIGH | NanoBanana는 비교적 새로운 서비스. SLA 보장 수준 미확인 |
| 가격 변동 | MEDIUM | 현재 ~$0.02/장이지만 가격 인상 가능성 |
| 이미지 일관성 | HIGH | AI 생성 이미지의 스타일 일관성 확보가 어려움. 같은 프롬프트도 매번 다른 결과 |
| 캐릭터 부재 | MEDIUM | 청월당처럼 고유 캐릭터를 만들 수 없다. "월하선생"의 시각적 정체성이 매번 달라짐 |

**조치**:
- DESIGN.md에 적힌 "ImageGenerator trait 추상화"는 좋은 결정. 폴백을 DALL-E 3 또는 Stable Diffusion으로 확보
- "월하선생" 캐릭터는 NanoBanana가 아닌 고정 일러스트로 제작 (앱 내 아이콘, 채팅 프로필)
- NanoBanana 스타일 시드 테스트를 **지금 즉시** 실행하여 품질/일관성 확인

### 8-3. Claude API 의존성 [MEDIUM]

| 리스크 | 심각도 | 상세 |
|--------|--------|------|
| 비용 폭증 | HIGH | 유료 해석 + 무료 운세 + 채팅의 합산 비용이 예상보다 높을 수 있음 |
| API 장애 | MEDIUM | 폴백(GPT-4o)이 설계되어 있으므로 경감됨 |

**조치**:
- 모델 계층화: Opus (유료 해석) / Sonnet (채팅) / Haiku (오늘의 운세)
- 비용 모델 재산정을 **최우선으로** 완료

### 8-4. 아키텍처 복잡도 [HIGH]

1인 또는 소규모 팀이 동시에 관리해야 하는 기술 스택:

## 최종 판정

**진행 가능하지만, 현재 계획대로는 위험하다.**

이 프로젝트의 핵심 가치(AI 대화형 사주 상담)는 유효하다. 시장도 크고, 빈틈도 있다. 하지만 실행 계획이 **과도한 기술적 야망**과 **불충분한 사업적 검증**의 조합이다. Rust 만세력 엔진은 기술적으로 아름답지만 1인 팀에게는 독이다. 앱 전용 + 웹 없음은 유입의 벽이다. 1회성 결제 모델은 LTV의 천장이다.

**권장 방향**:
1. "완벽한 엔진을 가진 앱"이 아니라 "빠르게 출시하여 시장 반응을 보는 MVP"로 전환
2. 만세력은 외부 소스, LLM은 모델 계층화, 웹 랜딩은 Phase 1에 추가, 구독은 v1에 포함
3. 3개월 안에 출시하고, 6개월 안에 PMF를 찾거나 피벗
